# Panda Island Windows MVP 开发路线图

## 1. 开发原则

- 每个阶段只解决一个可验收问题，不跨阶段堆功能。
- 先写纯逻辑测试，再实现状态机等核心模块。
- 原生窗口行为必须在 Windows 实机验证，不能只根据代码判断。
- 每阶段输出改动摘要、测试记录、构建结果、已知问题和回滚方式。
- 未通过阶段门禁，不进入下一阶段。

## 2. 阶段划分

### Stage 0：产品与架构确认（当前阶段）

交付物：

- `docs/PRD.md`
- `docs/ARCHITECTURE.md`
- `docs/DATA_MODEL.md`
- `docs/ROADMAP.md`
- `docs/TEST_PLAN.md`

完成标准：产品边界、Windows 目标、技术栈、数据模型、状态机、风险和验收标准不存在明显冲突；核心产品问题已确认并同步到全部文档。

### Stage 1：Tauri 项目骨架与窗口技术验证

范围：

- 初始化 Tauri 2 + React + TypeScript + Vite；
- ESLint、Prettier、Vitest、React Testing Library；
- 严格 TypeScript；
- 创建 `pet` 和 `panel` 两个窗口；
- 透明、无边框、360 × 360、置顶；
- 静态透明熊猫占位图；
- 拖动、右键菜单、托盘和退出；
- 点击穿透 Spike 和回退结论；
- Windows x64 开发构建。

门禁：实机验证透明、拖动、右键、托盘、资料窗口生命周期和真正退出；类型检查、单测、Rust 检查和 Tauri build check 通过。

### Stage 2：行为状态机

范围：

- `idle`、`walk`、`sleep`、`eat`、`react`、`dragged`；
- 可注入 Clock、RandomSource；
- 暂停/恢复；
- generation token 防止过期定时器；
- 先测试后实现；
- 暂用颜色块或静态帧表现状态，不接正式美术。

门禁：转移、边界时长、打断、暂停、重复调度和清理测试全部通过。

### Stage 3：PixiJS 动画播放器

范围：

- 载入本地动画 manifest 和图集；
- 状态到 clip 映射；
- 左右方向；
- 统一脚底锚点；
- 资源预加载和缺失降级；
- 与状态机完全解耦；
- 一套“花花”测试角色的原创动作，参考公开体态特征但不复制照片或官方形象；
- 点击反馈音、吃竹子音和轻量环境音，默认音量 35。

门禁：六种状态无跳位、无明显闪帧；连续切换不泄漏纹理；空闲资源占用达到 PRD 目标或记录偏差。

### Stage 4：资料卡与本地数据

范围：

- `pandas.json`、Zod schema 和 fixtures；
- 资料卡字段、来源、照片、收藏；
- 来源确认后的公开直播外链；
- HTTPS 和域名白名单；
- 简体中文完整；英文不进入 MVP。

门禁：非法数据和非法 URL 被拒绝；关闭资料卡不退出；重复打开只存在一个窗口。

### Stage 5：设置与持久化

范围：

- 熊猫选择、音量、静音、置顶、动画开关；
- 开机启动；
- 窗口位置、多显示器可见性校正；
- 设置版本和迁移；
- 恢复默认值。

门禁：重启后设置恢复；损坏设置可恢复；系统开机启动状态与界面一致。

### Stage 6：Windows 稳定性与发布候选版

范围：

- Windows 10/11、DPI、多显示器、集显/独显测试；
- 8 小时稳定性观察；
- 安装、升级、卸载；
- NSIS x64 安装包；
- 第三方许可清单；
- 签名策略和 SmartScreen 说明；
- 用户操作说明与验收报告。

门禁：P0 验收项全部通过，无阻断级缺陷；安装包哈希、测试报告和已知问题齐全。

## 3. 依赖顺序

| 模块       | 依赖                            |
| ---------- | ------------------------------- |
| 窗口与托盘 | Tauri 骨架                      |
| 行为状态机 | 无原生依赖，可在骨架后独立开发  |
| 动画播放器 | 状态类型、动画 manifest         |
| 资料卡     | 数据 schema、panel 窗口         |
| 外链       | 资料卡、capability、URL 白名单  |
| 设置       | Store 适配、窗口服务、Autostart |
| 打包       | 所有 P0 功能和资源授权清单      |

```text

构建产物：

已知问题：
回滚方法：
是否满足阶段门禁：
```

## 5. 缺陷等级

| 等级     | 定义                                  | 示例                                 |
| -------- | ------------------------------------- | ------------------------------------ |
| Blocker  | 无法启动、无法退出、数据/系统安全风险 | 应用启动即崩溃、退出后进程仍持续运行 |
| Critical | P0 主路径不可用                       | 无法拖动、资料卡打不开、设置无法保存 |
| Major    | 明显影响体验但有绕行                  | DPI 下偏位、状态动画偶发闪烁         |
| Minor    | 不阻断使用                            | 文案、轻微布局、低频视觉问题         |

发布候选版不允许存在 Blocker 或 Critical；Major 必须有明确接受人和修复计划。

## 6. 资源与内容前置条件

在 Stage 3 前必须准备：

- “花花”测试角色的原创 idle、walk、sleep、eat、react、dragged 素材；
- 统一画布、锚点、帧率和命名规范；
- 角色资源授权记录。
- 经过授权或原创的点击、进食和轻量环境音效，并记录来源。

在 Stage 4 前必须准备：

- 至少 1 条完整模拟熊猫数据；
- 占位照片；
- 若测试直播跳转，提供一个允许公开访问的 HTTPS 测试链接和来源名称。

## 7. 后续版本候选，不进入 MVP

- Windows ARM64；
- 自动更新和签名发布流水线；
- 多只真实熊猫审核资料；
- Supabase 内容后台；
- 通知和熊猫日记；
- AI 百科问答（仅基于审核资料的 RAG）；
- macOS 版本；
- Spine 或 3D 角色系统。
